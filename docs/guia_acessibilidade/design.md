### Usabilidade, IHC e Acessibilidade

* A usabilidade deve permitir que o usuário atinja seus objetivos de forma eficaz, eficiente e satisfatória; quando a interface cria barreiras para pessoas com deficiência, ela falha também como experiência de uso. [NBR 5.1.13] [WCAG 2.2 - Princípios Básicos]

* A interface deve ser operável, compreensível e previsível, pois a IHC busca reduzir erros, aumentar eficiência e melhorar a satisfação do usuário. [NBR 5.7.15] [WCAG 2.2 - 3.2 Previsível]

* A acessibilidade digital deve ser tratada como parte central da qualidade da interface, não como funcionalidade opcional. [NBR 4] [WCAG 2.2 - Introdução]

* Processos de login e autenticação não devem forçar o usuário a resolver testes cognitivos complexos (como transcrever textos distorcidos de imagens ou memorizar dados), permitindo métodos alternativos seguros, como preenchimento automático por gerenciadores de senha ou biometria. [WCAG 2.2 - 3.3.8 Autenticação Acessível (Mínimo)]

### Design Visual, Contraste e Cores

* A interface não deve depender apenas da cor para transmitir informações, como erro, sucesso, alerta ou estado selecionado. [NBR 5.11.1] [WCAG 2.2 - 1.4.1 Uso de Cor]

* Textos devem possuir contraste mínimo suficiente em relação ao fundo, garantindo legibilidade para usuários com baixa visão. [NBR 5.11.3] [WCAG 2.2 - 1.4.3 Contraste (Mínimo)]

* Componentes de interface, como botões, campos e bordas, devem ter contraste adequado para serem percebidos visualmente. [NBR 5.11.4] [WCAG 2.2 - 1.4.11 Contraste Não Textual]

* Objetos gráficos importantes, como ícones, indicadores e elementos informativos, devem possuir contraste suficiente. [NBR 5.11.5] [WCAG 2.2 - 1.4.11 Contraste Não Textual]

* O indicador de foco também deve possuir contraste perceptível, para que o usuário consiga acompanhar a navegação por teclado. [NBR 5.11.6] [WCAG 2.2 - 1.4.11 Contraste Não Textual, 2.4.13 Aparência do Foco]

---

### Apresentação, Layout e Responsividade

* A ordem de apresentação dos elementos deve preservar o significado do conteúdo, mantendo uma sequência lógica para leitura e interação. [NBR 5.10.2] [WCAG 2.2 - 1.3.2 Sequência com Significado]

* O design deve ser responsivo, funcionando adequadamente em diferentes tamanhos de tela e dispositivos. [NBR 5.10.4] [WCAG 2.2 - 1.4.10 Refluxo]

* A interface não deve depender apenas de características sensoriais, como posição, formato, tamanho ou cor, para explicar uma ação. [NBR 5.10.1] [WCAG 2.2 - 1.3.3 Características Sensoriais]

* O zoom não deve ser bloqueado, permitindo que o usuário amplie o conteúdo conforme sua necessidade. [NBR 5.13.5] [WCAG 2.2 - 1.4.4 Redimensionamento de Texto]

* O texto deve poder ser redimensionado sem perda de conteúdo ou funcionalidade. [NBR 5.12.7] [WCAG 2.2 - 1.4.4 Redimensionamento de Texto]

### Relação com “Bad vs Good UI Design”

* Uma interface “good UI” deve tornar as ações visíveis, compreensíveis e previsíveis, reduzindo a necessidade de instruções excessivas para tarefas simples. [NBR 5.8.3] [WCAG 2.2 - 3.3.2 Rótulos ou Instruções]

* A consistência visual e funcional contribui para que o usuário reconheça padrões de navegação e interação ao longo da interface. [NBR 5.7.15] [WCAG 2.2 - 3.2.3 Navegação Consistente, 3.2.4 Identificação Consistente]

* A visibilidade do estado do sistema deve ser garantida por mensagens, foco visível, feedback de ações e identificação clara dos componentes. [NBR 5.1.1] [NBR 5.13.8] [WCAG 2.2 - 4.1.3 Mensagens de Status]

* O bom design não deve apenas “parecer bonito”; ele deve permitir uso eficiente, acessível e seguro para diferentes perfis de usuários. [NBR 5.10.4] [WCAG 2.2 - Princípio 4 Robusto]
